浏览器请求+渲染流程

在开发前端过程中，无论是优化还是想设计高性能页面，浏览器作为网页的载体，这其中的 网络流程、页面渲染过程，JavaScript 执行流程，以及 Web 安全理论都是我们必须要了解的， 因此，学习浏览器的多进程架构是很有必要的。

进程和线程

以Chrome为例，我们打开任务管理器，可以看到浏览器中启动的进程个数超过我们打开页面的个数，那么这些进程到底有什么用，在页面渲染，网络请求中各自担当什么角色呢？

并行处理

在介绍一下进程和线程的概念之前，我们简单了解一下什么叫并行处理。

并行处理就是同时处理多个任务，就比如上面三个式子，若是使用单线程来处理，那么就是分四步按顺序执行四个任务，但如果是多线程，只需要分两步， 第一步，使用三个线程同时执行前三个任务；第二步，再执行第四个显示任务。因此，使用并行处理能大大提升性能。

线程与进程

那么现在开始介绍线程和进程。通俗一点来讲，进程好比火车，线程是火车的车厢，线程不能独立存在，必须由进程来启动和管理。

那么进程到底是什么？详细一点来讲， 启动一个程序的时候，操作系统会为该程序创建一块内存，用来存放代码、运行中的数据和一个执行任务的主线程，我们把这样的一个运行环境叫进程。而线程中可以处理进程中的小任务。

看这张图就很明显可以明白为什么多线程处理任务能提升运算效率啦。

稍微总结一下，进程和线程之间的关系有以下 4 个特点：

  1. 进程中任意一个线程执行出错，都会导致整个进程的崩溃。
  2. 线程中共享进程中的数据，可以对公共数据进行读写操作
  3. 进程之间的数据互相无法访问
  4. 当一个进程结束，操作系统会回收进程所占用的内存

多进程浏览器架构

那么如今的浏览器进程架构是怎样的呢？ 最新的 Chrome 浏览器包括：1 个浏览器（Browser）主进程、1 个 GPU 进程、1 个网络（NetWork）进程、多个渲染进程和多个插件进程。

浏览器进程主要是负责页面显示，用户交互，子进程通信管理和存储 localStorage,cookie 之类的。

GPU 进程是在页面UI渲染的时候会使用GPU加速渲染和3Dcss效果

网络进程涉及到下载网络资源，向服务器发送请求是在网络进程中完成。

渲染进程将网络进程请求返回的HTML、CSS 和 JavaScript 转换为用户可以与之交互的网页。

插件进程顾名思义就是负责插件的运行啦。

后面两个渲染和插件进程都是运行在沙箱模式中。什么叫沙箱模式？简单来说就是操作系统给进程上的锁，程序可以运行，但是不能对硬盘写入任何数据，主要是为了阻止执行恶意程序对电脑产生影响，比如病毒，窃取信息啥的。

对比起早期浏览器单线程时期（所有功能模块都运行在同一进程）这样的改变有什么好处呢？

第一每个进程都是相互隔离的，一个页面拥有单独的进程（渲染），如果它崩溃了不会影响整个浏览器或者其他页面（但如果是同一域名下打开的多个网页还是会影响哈， 官方把这个默认策略叫process-per-site-instance， 如果几个页面符合同一站点，那么他们将被分配到一个渲染进程里面去）相比起来更加稳定。

第二就是在单线程时代，有时候运行一个复杂点的页面再关闭，内存是无法完全回收的（名词叫做页面的内存泄漏），导致浏览器使用时间越长，内存被占用得越多，浏览器就变得更慢。线程的关闭是不会回收内存的。多进程解决内存泄漏的问题就很简单了， 因为当关闭一个页面时，整个渲染进程也会被关闭，之后该进程所占用的内存都会被系统回收。

从输入 URL 到页面展示

从输入URL到渲染出完整页面，其中涉及到对于网络协议流程的理解，HTTP和 TCP/IP协议是必须要了解的一环，那么其中的流程到底是什么样的，简单串一下。

其实本来的流程应该是这样的：

  1. 用户输入，浏览器根据输入内容判断是URL还是搜索内容，若是URL补齐协议名后构建请求发送网络请求。若是搜索内容，转变成关键字进行搜索。 浏览器还给了当前页面一次执行 beforeunload 事件的机会，允许页面在退出之前执行一些数据清理操作，还可以询问用户是否要离开当前页面

  2. 浏览器进程会通过进程间通信（IPC）把 URL 请求发送至网络进程。网络进程在真正发送请求之前，先在浏览器中查找缓存，包括资源副本和DNS数据缓存，若有副本那么会拦截请求，返回副本结束请求。（咱就是说这里又涉及到协商缓存和强缓存，重点但是这里就不介绍了）

  3. 若没有副本，那么根据URL请求DNS返回域名IP以及端口。HTTP默认端口80，HTTPS默认443。 如果请求协议是 HTTPS，那么还需要建立 TLS 连接。

  4. 等待TCP队列（可选，若是同一域名内少于6个TCP连接，则不用等待）

  5. 建立TCP连接，三次握手。

  6. 发送HTTP请求，传输HTTP数据，服务器返回数据包。相应数据顺着应用层——传输层——网络层——网络层——传输层——应用层的顺序。

  7. 紧接着网络进程接收响应头数据，解析后把数据发给浏览器进程。浏览器发送“提交导航”到渲染进程。这里还涉及到重定向 （301 或者 302）和相应数据类型处理（Content-Type）的选择。

  8. 渲染进程开始准备接受 HTML 数据，接收数据的方式是直接和网络进程建立数据管道（使用IPC建立进程间通讯）然后传输数据。

  9. 传输完成，渲染进程告诉浏览器“确认提交”，浏览器会更新浏览器界面状态，包括了安全状态、地址栏的 URL、前进后退的历史状态，并更新 Web 页面（ 此时的web页面是空白页 ）。

  10. 渲染进程开始对文档进行页面解析和资源加载，这其中也有点复杂，后面再详细讲。绘制好页面后，最后新页面显示在屏幕中。

这大概就是“从输入 URL 到页面展示，这中间发生了什么？”的答案，当然其中还有一些细节 。

那么接下来我们从第一步开始细致说说，浏览器到底都干了些什么事。

网络协议

互联网，实际上是一套理念和协议组成的体系架构 。网络资源请求是个很复杂的流程，涉及到各种协议的使用，所以干脆就从网络协议入手，从“数据包如何送达主机”,“数据如何被完整的送达应用程序”两个角度来讲述数据的传输过程。

IP协议（ 网际协议 ）

互联网上不同的设备都有唯一的地址，这个地址就是一串数字， 计算机的地址就称为 IP 地址。

我们访问网站，其实就是你的计算机向另外一台计算机请求信息。我们可以类比成快递：A给B邮寄东西，不但要写B的地址，还要附加A的信息，这和网络差不多。所以数据包上其实是有两个IP地址的，一个是你的计算机的，一个是网址解析成的IP。

IP协议负责的就是数据包如何送达主机这一部分， IP是整个TCP/IP协议族的核心，也是构成互联网的基础。它位于TCP/IP模型的网络层。

再具体下去就比较复杂了，只介绍到这里。 IP 是非常底层的协议，只负责把数据包传送到对方电脑。

UDP协议

既然说到TCP，那也顺便说一说UDP协议。他俩都是传输层的协议，但是区别还是很大的。

当数据包传送到电脑，基于IP协议之上与应用打交道的协议就是UDP和TCP了。

UDP 中一个最重要的信息是端口号。IP 通过 IP 地址信息把数据包发送给指定的电脑，而 UDP 通过端口号把数据包分发给正确的程序。

但是UDP这个东西，它很快但是不能保证数据被完整的传输，可能会丢包或者出错，也不提供重发机制，所以一般应用在关注速度的领域（在线视频之类）。并且大文件在传输的时候会被拆解成小包来传输，不同路径不同时间，传到的时候UDP也不知道怎么组装，还原不了完整文件。

为了解决这些问题，所以我们引入了TCP协议，它安全可靠还提供序列号，但是速度慢。

TCP协议

一个完整的 TCP 连接的生命周期包括了“建立连接”“传输数据”和“断开连接”三个阶段。

三次握手：第一次：客户端给服务端发送一个带有SYN （同步序列号） 标志的数据包 第二次：服务端给客户端发送带有SYN和ACK（确认应答）标志得数据包 第三次：客户端给服务端发送带有ACK标志的数据包。（ 为什么是三次？ 一次肯定不行，客户端发过去之后服务端无响应，客户端就不知道是否可以进行数据数据传递 两个也不行，比如客户端给服务端说我要进行传递数据，服务端说可以传递，两个握手完成。如果没有第三次客户端的确认，服务端不知道客户端是否收到它允许传递的信号，所以两次也不行。 ）

其次传输数据，会对数据包逐个确认，也就是接收端会发送确认数据包到发送端，若规定时间内没收到确认包，判断为丢失，则重发。

HTTP协议

HTTP 协议，正是建立在 TCP 连接基础之上的。因为浏览器使用 HTTP 协议作为应用层协议，用来封装请求的文本信息向服务器获取资源；并使用 TCP/IP 作传输层协议将它发到网络上。具体的学习HTTP还是看图解HTTP吧，这里就简单说一下。

在 HTTP 工作开始之前，浏览器需要通过 TCP 与服务器建立连接。这也是HTTP网络请求的第一步。也就是说 HTTP 的内容是通过 TCP 的传输数据阶段来实现的。

首先浏览器会向服务器发送请求行，它包括了请求方法、请求 URI（Uniform Resource Identifier）和 HTTP 版本协议。 like this：
