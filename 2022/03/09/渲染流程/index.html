渲染流程

接上一话，导航被提交以后的渲染流程是怎样工作的。渲染阶段至关重要，关乎优化页面卡顿问题。这篇博客介绍从网络流程向渲染流程传输HTML，CSS和js文件到整个页面完全被渲染出来，背后的原理和一整条流水线。

按照渲染的时间顺序，流水线可分为如下几个子阶段：构建 DOM 树、样式计算、布局阶段、分层、绘制、分块、光栅化和合成。

构建DOM树

为什么要构建DOM树呢？因为浏览器没办法直接理解和使用html，所以要通过浏览器HTML解析器，把它转换成浏览器能理解的结构，也就是DOM树了。

从表象来看，DOM结构和HTML内容一样，但是DOM不同在于它是保存在内存中的树状结构，可以通过js来查询和修改其内容。

样式计算

样式计算的目的是为了计算出 DOM 节点中每个元素的具体样式。

1.转换css

css就好像html文件一样，浏览器无法直接理解，所以在渲染流程接受css文件之前，会将css文件转换成 styleSheets以供浏览器理解。可以通过 document.styleSheets访问到，也可以被修改和查询。

2.转换属性值

CSS 文本中有很多属性值，如 2em、blue、bold，这些类型数值不容易被渲染引擎理解，所以需要将所有值转换为渲染引擎容易理解的、标准化的计算值，这个过程就是属性值标准化。

被转换成px，rgb表示的颜色等等

3.计算出每个节点的具体样式

属性都已经标准化了，接下来就只需要对应到每个节点了。这里就涉及到 CSS 的继承规则和层叠规则。

所谓继承规则，举个栗子,，比如开发者在网页中设置p段落标签，但是在css中并没有设置该标签的对应样式，但是我们在浏览器中仍可以看到这个段落文字。那么这里的文字大小颜色，margin值等等一系列属性到底是哪里来的呢？来源于默认样式和对父节点样式的继承。

这就是最简单的继承规则的应用，假设这个p标签的路径是 html.body.div.p，那么p这个DOM节点就包含了父节点的样式。 这里需要特别提下 UserAgent 样式，它是浏览器提供的一组默认样式，如果你不提供任何样式，默认使用的就是 UserAgent 样式。

第二个规则是层叠规则。 层叠是 CSS 的一个基本特征，它是一个定义了如何合并来自多个源的属性值的算法。 当你随机打开一个页面，查看源代码，你会发现，同一个元素，不止有一个CSS选择器及对应的样式。而一个元素只能应用一个样式，那么一堆样式中究竟是应用哪一个呢？

首先样式的来源分为五种。

1、浏览器默认的样式；

2、用户自定义样式。一些页面中会提供一些让用户自定义字体大小颜色等的快捷键；

3、外部样式，即引用的CSS后缀文件；

4、内部样式，即写在标签内的样式；

5、内联样式，即直接写在style属性内的样式

其中后三种都可以被归为开发者设定。 层叠规则如下：

  1. 按显式权重对应用到该元素的所有声明进行排序，有!important标志的声明的权重要高于不带!important标志的声明。
  2. 开发者的样式和读者样式斗都比用户代理的默认样式强。
  3. 按特殊性对应用到给定元素的所有声明进行排序，有较高特殊性的元素权重要大于有较低特殊性的元素。
  4. 按出现顺序对应用到给定元素的所有声明进行排序，后面出现的声明权重要大于前面出现的声明，即后定义的样式会覆盖前面定义的样式。

其中第三点 选择器的特殊性 很重要它们满足这样的优先级：

1、对于内联样式，特殊性首位加1，即1，0，0，0。

2、对于选择器中出现的ID属性值，加0，1，0，0， 有多少个ID值就在第二位加几位。

3、对于选择器中出现的类属性值，属性选择及伪类，加0，0，1，0，共出现多少个就在第三位加几位。

4、对于选择器中出现的元素，以及伪元素，加0，0，0，1，共出现多少个就在第四位加几位。

样式计算阶段遵守 CSS 的继承和层叠两个规则计算出 DOM 节点中每个元素的具体样式。

布局阶段

有了 DOM 树和 DOM 树中元素的样式， 我们还需要知道 DOM 元素的几何位置信息。那么接下来就需要计算出 DOM 树中可见元素的几何位置，我们把这个计算过程叫做布局。 在布局阶段需要完成两个任务：创建布局树和布局计算。

创建布局树

浏览器大体上完成了下面这些工作：

  1. 遍历 DOM 树中的所有可见节点，并把这些节点加到布局树中；
  2. 而不可见的节点会被布局树忽略掉，如 head 标签下面的全部内容，再比如 body.p.span 这个元素，因为它的属性包含 dispaly:none，所以这个元素也没有被包进布局树。

布局计算

布局计算也是个非常复杂的东西，这里就不展开讲，毕竟主要是讲渲染流程，反正布局计算的结果计算好，也就是每个元素的几何坐标位置，以后会被重新写到布局树里面。

分层

在开始绘制页面之前，因为页面中有很多复杂的效果，如一些复杂的 3D 变换、页面滚动，或者使用 z-indexing 做 z 轴排序等，为了更加方便地实现这些效果，渲染引擎还需要为特定的节点生成专用的图层，并生成一棵对应的图层树。

浏览器的页面实际上被分成了很多图层，这些图层叠加后合成了最终的页面。但不是每个节点都是一个图层 如果一个节点没有对应的层，那么这个节点就从属于父节点的图层。

渲染引擎会为哪些节点创建新的图层？：

1.拥有层叠上下文属性的元素 层叠上下文属性：明确定位属性的元素（position），定义透明属性的元素（opacity），使 用css滤镜元素（filter），设置z-index属性的元素。

2.需要裁剪的地方。就比如我们定义了一个div，但是div里面的文字超出我们定义div的面积，那么就会产生剪裁，渲染引擎会把裁剪文字内容的一部分用于显示在 div 区域，或者出现滚动条，也会被提升为单独的一层。

图层绘制

说了半天终于来到了绘制的阶段，渲染引擎会对图层树中的每个图层进行绘制。渲染引擎会把 一个图层的绘制拆分成很多小的绘制指令，然后再把这些指令按照顺序组成一个待绘制列表，按照列表去绘制每一个元素。

栅格化（raster）操作

实际上绘制操作是由渲染引擎中的合成线程来完成的。 渲染进程中的主线程将绘制列表传递给合成线程。

通常一个页面可能很长，但是用户只能看到一部分，这一部分就是视口。如果一次性绘制所有图层所有内容，开销太大也没有必要。所以合成线程需要图层划分为图块，按照视口附近的图块来优先生成位图。

而实际生成位图的操作是由栅格化来执行的。所谓栅格化，是指将图块转换为位图。渲染进程维护了一个栅格化的线程池，所有的图块栅格化都是在线程池内执行的。 通常，栅格化过程都会使用 GPU 来加速生成，使用 GPU 生成位图的过程叫快速栅格化，或者 GPU 栅格化，生成的位图被保存在 GPU 内存中。

合成和显示

一旦所有图块都被光栅化，合成线程就会生成一个绘制图块的命令——“DrawQuad”，然后将该命令提交给浏览器进程。

浏览器进程里面有一个叫 viz 的组件，用来接收合成线程发过来的 DrawQuad 命令，然后根据 DrawQuad 命令，将其页面内容绘制到内存中，最后再将内存显示在屏幕上。

到这里，渲染流程就结束啦。

总结

整体流程放在这里。

结合上图，一个完整的渲染流程大致可总结为如下：

  1. 渲染进程将 HTML 内容转换为能够读懂的 DOM 树结构。
  2. 渲染引擎将 CSS 样式表转化为浏览器可以理解的 styleSheets，计算出 DOM 节点的样式。
  3. 创建布局树，并计算元素的布局信息。
  4. 对布局树进行分层，并生成分层树。
  5. 为每个图层生成绘制列表，并将其提交到合成线程。
  6. 合成线程将图层分成图块，并在光栅化线程池中将图块转换成位图。
  7. 合成线程发送绘制图块命令 DrawQuad 给浏览器进程。
  8. 浏览器进程根据 DrawQuad 消息生成页面，并显示到显示器上。

这里再多说一下相关概念：

重绘，重排和合成

  1. 更新了元素的几何属性（重排） 例如改变元素的宽度、高度等，那么浏览器会触发重新布局，解析之后的一系列子阶段，这个过程就叫重排。无疑，重排需要更新完整的渲染流水线，所以开销也是最大的。

  2. 更新元素的绘制属性（重绘） 比如通过 JavaScript 更改某些元素的背景颜色。 那么布局阶段将不会被执行，因为并没有引起几何位置的变换，所以就直接进入了绘制阶段，然后执行之后的一系列子阶段，这个过程就叫重绘。 重绘省去了布局和分层阶段，所以执行效率会比重排操作要高一些。

  3. 直接合成阶段（更改一个既不要布局也不要绘制的属性） 这样的效率是最高的，因为是在非主线程上合成，并没有占用主线程的资源，另外也避开了布局和绘制两个子阶段

    减少重排重绘, 方法很多：

  4. 使用 class 操作样式，而不是频繁操作 style

  5. 避免使用 table 布局

  6. 批量dom 操作，例如 createDocumentFragment，或者使用框架，例如 React

  7. Debounce window resize 事件

  8. 对 dom 属性的读写要分离

  9. will-change: transform 做优化
