栈空间和堆空间

语言类型

首先需要认知一下js是一种什么类型的语言？

一般来说代码声明有两种类型，一种是在声明变量之前要定义 变量类型。我们把这种在使用之前就需要确认其变量数据类型的称为静态语言。相反地，我们把在运行过程中需要检查数据类型的语言称为动态语言。比如 JavaScript 就是动态语言，因为在声明变量之前并不需要确认其数据类型。

同时还分为强类型和弱类型的区别，支持隐式类型转换的语言称为弱类型语言，不支持隐式类型转换的语言称为强类型语言。啥是隐式转换呢？就是说可以将变量的类型不断变换，比如 int 型的变量悄悄转换为 bool 型的变量。

js作为动态弱类型语言，在运行过程中数据是如何存储的呢？

存储空间类型

在 JavaScript 的执行过程中， 主要有三种类型内存空间，分别是代码空间、栈空间和堆空间。代码空间是用来存储可执行代码之类的东西，而栈空间是用来存储执行上下文的，而执行上下文中又包括变量环境和词法环境，堆空间则是用来存储对象之类的引用类型的数据。

也就是说原始类型的数据值都是直接保存在“栈”中的，引用类型的值是存放在“堆”中的。在栈中只会保留引用类型的一个引用地址用来访问该数据。

那为什么要分两种存储空间呢？主要是因为 JavaScript 引擎需要用栈来维护程序执行期间上下文的状态，如果栈空间大了话，所有的数据都存放在栈空间里面，那么会影响到上下文切换的效率，进而又影响到整个程序的执行效率。js中的各种方法程序嵌套调用都涉及到栈来操作，所以栈对于js来说是不可或缺的数据类型。举个栗子：比如 foo 函数执行结束了，JavaScript 引擎需要离开当前的执行上下文，只需要将指针下移到上个执行上下文的地址就可以了，foo 函数执行上下文栈区空间就可以被全部回收，这又涉及到垃圾回收策略了。

所以通常情况下，栈空间都不会设置太大，主要用来存放一些原始类型的小数据。而引用类型的数据占用的空间都比较大，所以这一类数据会被存放到堆中，堆空间很大，能存放很多大的数据，不过缺点是分配内存和回收内存都会占用一定的时间。

还有两个小tips： 在 JavaScript 中，赋值操作和其他语言有很大的不同，原始类型的赋值会完整复制变量值，而引用类型的赋值是复制引用地址。 所以这里又涉及到深复制和浅复制的关系，在复制引用类型的时候要记得用for循环或者其他遍历方法复制，如果只是简单的=，那么只会复制一个堆内存地址哦。

闭包是存储在堆空间的对象哦，是一个内部对象，js是无法访问的。

垃圾回收

js中使用的是自动垃圾回收的策略，产生的垃圾数据是由垃圾回收器来释放的，并不需要手动通过代码来释放（c就是手动的）。接着上面的说， 分别介绍“栈中的垃圾数据”和“堆中的垃圾数据”是如何回收的。

堆中的数据是如何回收的

要回收堆中的垃圾数据，就需要用到 JavaScript 中的垃圾回收器了。

那么首先要先了解一下代际假说（The Generational Hypothesis）的内容，这是垃圾回收领域中一个重要的术语，后续垃圾回收的策略都是建立在该假说的基础之上的，所以很重要。

代际假说有以下两个特点：

第一个是大部分对象在内存中存在的时间很短，简单来说，就是很多对象一经分配内存，很快就变得不可访问；

第二个是不死的对象，会活得更久。

基于代际假说，V8 将堆内存分为两个部分，分别是新生代和老生代两个区域，新生代中存放的是生存时间短的对象，老生代中存放的生存时间久的对象。 V8 分别使用两个不同的垃圾回收器 ：副垃圾回收器，主要负责新生代的垃圾回收。主垃圾回收器，主要负责老生代的垃圾回收。但是不管是什么类型的垃圾回收器，它们都有一套共同的执行流程。

第一步是标记空间中活动对象和非活动对象。所谓活动对象就是还在使用的对象，非活动对象就是可以进行垃圾回收的对象。

第二步是回收非活动对象所占据的内存。其实就是在所有的标记完成之后，统一清理内存中所有被标记为可回收的对象。

第三步是做内存整理。一般来说，频繁回收对象后，内存中就会存在大量不连续空间，我们把这些不连续的内存空间称为内存碎片。当内存中出现了大量的内存碎片之后，如果需要分配较大连续内存的时候，就有可能出现内存不足的情况。所以最后一步需要整理这些内存碎片，但这步其实是可选的。

副垃圾回收器

通常情况下，大多数小的对象都会被分配到新生区，所以说这个区域不大，但是垃圾回收还是比较频繁的。

新生代中涉及到 Scavenge 算法 ：是把新生代空间对半划分为两个区域，一半是对象区域，一半是空闲区域。

新加入的对象都会存放到对象区域，当对象区域快被写满时，就需要执行一次垃圾清理操作。这个过程就是，先给垃圾做标记，完成后把那些活跃的对象复制到空闲区域并有序排列，之后对对象区域中的数据经常删除，然后对象区域与空闲区域进行角色翻转，这样就完成了一次清除。但是因为新生区空间不大，所以为了减少垃圾清除的频率， JavaScript 引擎采用了对象晋升策略，也就是经过两次垃圾回收依然还存活的对象，会被移动到老生区中。

主垃圾回收器

除了新生区中晋升的对象，一些大的对象会直接被分配到老生区。因此老生区中的对象有两个特点，一个是对象占用空间大，另一个是对象存活时间长。 主垃圾回收器是采用标记 - 清除（Mark-Sweep）的算法进行垃圾回收的。

首先还是标记阶段，遍历调用栈，遍历没有到达的地方就是垃圾数据。 接下来就是垃圾的清除过程。

上面的标记过程和清除过程就是标记 - 清除算法，不过对一块内存多次执行标记 - 清除算法后，会产生大量不连续的内存碎片。而碎片过多会导致大对象无法分配到足够的连续内存，于是又产生了另外一种算法——标记 - 整理（Mark-Compact），这个标记过程仍然与标记 - 清除算法里的是一样的，但后续步骤不是直接对可回收对象进行清理，而是让所有存活的对象都向一端移动，然后直接清理掉端边界以外的内存。

全停顿

JavaScript 是运行在主线程之上的，一旦执行垃圾回收算法，都需要将正在执行的 JavaScript 脚本暂停下来，待垃圾回收完毕后再恢复脚本执行。我们把这种行为叫做全停顿。

为了降低老生代的垃圾回收而造成的卡顿，V8 将标记过程分为一个个的子标记过程，同时让垃圾回收标记和 JavaScript 应用逻辑交替进行，直到标记阶段完成，我们把这个算法称为增量标记（Incremental Marking）算法。这样就不会让用户因为垃圾回收任务而感受到页面的卡顿啦。

编译器和解释器

机器不能直接理解我们所写的代码，所以在执行程序之前，需要将我们所写的代码“翻译”成机器能读懂的机器语言。按语言的执行流程，可以把语言划分为编译型语言和解释型语言。

编译型语言在程序执行之前，需要经过编译器的编译过程，并且编译之后会直接保留机器能读懂的二进制文件，这样每次运行程序时，都可以直接运行该二进制文件，而不需要再次重新编译了。而由解释型语言编写的程序，在每次运行时都需要通过解释器对程序进行动态解释和执行。

大概的翻译流程是这样。

V8 是如何执行一段 JavaScript 代码的

  1. 生成抽象语法树（AST）和执行上下文

    将源代码转换为抽象语法树，并生成执行上下文。可以把 AST 看成代码的结构化的表示，编译器或者解释器后续的工作都需要依赖于 AST，而不是源代码。 Babel也是AST的一个应用，Babel 的工作原理就是先将 ES6 源码转换为 AST，然后再将 ES6 语法的 AST 转换为 ES5 语法的 AST，最后利用 ES5 的 AST 生成 JavaScript 源代码。

    生成AST有两个阶段：

    1. 第一阶段是分词（tokenize），又称为词法分析，其作用是将一行行的源码拆解成一个个 token。所谓 token，指的是语法上不可能再分的、最小的单个字符或字符串。
    2. 第二阶段是解析（parse），又称为语法分析，其作用是将上一步生成的 token 数据，根据语法规则转为 AST。

  1. 生成字节码

    此时解释器 Ignition 就登场了，它会根据 AST 生成字节码，并解释执行字节码。为啥不用机器码呢？因为机器码占用内存太大了，字节码是介于 AST 和机器码之间的一种代码。但是与特定类型的机器码无关，字节码需要通过解释器将其转换为机器码后才能执行。机器码所占用的空间远远超过了字节码，所以使用字节码可以减少系统的内存使用。

  2. 执行代码

    通常，如果有一段第一次执行的字节码，解释器 Ignition 会逐条解释执行。到了这里，相信你已经发现了，解释器 Ignition 除了负责生成字节码之外，它还有另外一个作用，就是解释执行字节码。在 Ignition 执行字节码的过程中，如果发现有**热点代码**（HotSpot），比如一段代码被重复执行多次，这种就称为热点代码，那么后台的编译器 TurboFan 就会把该段热点的字节码编译为高效的机器码，然后当再次执行这段被优化的代码时，只需要执行编译后的机器码就可以了，这样就大大提升了代码的执行效率。
